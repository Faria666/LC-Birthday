# LC-Birthday Static Site Constitution

## Core Principles

### I. Technology Stack (NON-NEGOTIABLE)

The site is a **zero-framework static site**: plain HTML5, vanilla CSS, and TypeScript compiled to ESM.
No React, Vue, Angular, Svelte, or any runtime component framework is permitted.
Third-party runtime JavaScript must be justified individually — every byte shipped is a deliberate decision.

Permitted external dependencies (build-time or CDN with SRI hash):
- **TypeScript** — compile only, nothing ships at runtime
- **Vite** — bundler/dev server; single source of truth for the build pipeline
- **Vitest** — unit and integration tests
- **ESLint + typescript-eslint** — linting
- **Prettier** — formatting (no custom rules that conflict with Prettier)
- **Stylelint** — CSS linting

### II. File & Folder Structure (NON-NEGOTIABLE)

```
/
├── src/
│   ├── index.html          # Entry point (only one per page)
│   ├── assets/             # Images, fonts, icons — optimised at build time
│   │   ├── images/
│   │   └── fonts/
│   ├── styles/
│   │   ├── tokens.css      # Design tokens (custom properties only)
│   │   ├── reset.css       # Modern CSS reset
│   │   ├── global.css      # Base typography & layout primitives
│   │   └── components/     # One file per logical UI component
│   ├── scripts/
│   │   ├── main.ts         # Single entry point; imports all modules
│   │   └── modules/        # One file per feature/concern
│   └── types/              # Shared TypeScript type declarations
├── public/                 # Static assets served verbatim (robots.txt, favicon, CNAME)
├── tests/
│   ├── unit/
│   └── e2e/
├── .github/
│   └── workflows/
│       └── deploy.yml      # GitHub Pages CI/CD
├── vite.config.ts
├── tsconfig.json
├── .eslintrc.json
├── .stylelintrc.json
├── .prettierrc
└── package.json
```

Any deviation from this structure requires a documented rationale in the PR.

### III. TypeScript Rules (NON-NEGOTIABLE)

`tsconfig.json` must enable **strict mode** at minimum:

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitOverride": true,
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "lib": ["ES2022", "DOM", "DOM.Iterable"],
    "outDir": "dist",
    "sourceMap": true,
    "declaration": true
  }
}
```

- `any` is **banned** — use `unknown` and narrow explicitly.
- Type assertions (`as X`) require a comment explaining why the compiler cannot prove the type.
- All exported functions must have explicit return types.
- DOM queries must be null-checked before use — no non-null assertion (`!`) on query results.

### IV. CSS Architecture (NON-NEGOTIABLE)

CSS is authored in plain CSS (no Sass/LESS). Architecture follows these rules:

1. **Design tokens first** — every colour, spacing, font size, radius, and shadow is a CSS custom property in `tokens.css`. Raw values (e.g., `#3b82f6`, `16px`) are banned outside `tokens.css`.
2. **BEM naming convention** — block, element, modifier. Example: `.card`, `.card__title`, `.card--featured`.
3. **No utility-class soup** — no Tailwind or inline style attributes (except generated by JS where unavoidable and documented).
4. **Cascade layers** — use `@layer reset, tokens, base, components, utilities` to control specificity intentionally.
5. **Logical properties** — prefer `margin-inline`, `padding-block`, `inset-inline-start` over physical `left`/`right`/`top`/`bottom` for internationalisation readiness.
6. **No `!important`** — a specificity problem is an architecture problem; fix the architecture.

### V. Accessibility (NON-NEGOTIABLE — WCAG 2.1 AA)

Every shipped UI element must meet WCAG 2.1 AA. This is not aspirational; it is a hard gate:

- All images have meaningful `alt` text (decorative images use `alt=""`).
- Colour contrast ratio ≥ 4.5:1 for normal text, ≥ 3:1 for large text and UI components.
- All interactive elements are reachable and operable by keyboard alone.
- Focus indicators are visible and meet contrast requirements.
- Semantic HTML is used in preference to `div` soup — `<nav>`, `<main>`, `<article>`, `<section>`, `<header>`, `<footer>`, `<button>`, `<a>` used correctly.
- ARIA is used only when native HTML semantics are insufficient. No ARIA role duplicates a native element's implicit role.
- Page has a single `<h1>`; heading hierarchy is logical and unbroken.
- Skip-to-main-content link is the first focusable element in the document.
- All form inputs are associated with a `<label>` (explicit `for`/`id` or wrapping label).
- `lang` attribute is set on `<html>`.

### VI. Performance (NON-NEGOTIABLE — Core Web Vitals targets)

Target scores on a mid-range mobile device on a 4G connection:
- **LCP** < 2.5 s
- **INP** < 200 ms
- **CLS** < 0.1
- **Lighthouse Performance score** ≥ 90

Rules:
- All images: `width` + `height` attributes set; use `loading="lazy"` for below-the-fold images; provide modern formats (WebP/AVIF) via `<picture>`.
- Fonts: `font-display: swap`; self-hosted or preloaded with `<link rel="preload">`.
- No render-blocking scripts — all `<script>` tags use `defer` or `type="module"`.
- HTTP cache headers: assets with content hash in filename get `Cache-Control: max-age=31536000, immutable` via GitHub Pages `_headers` file.
- No synchronous XHR. No `document.write`.
- JS bundle size for critical path: < 50 kB compressed.

### VII. SEO & Metadata (NON-NEGOTIABLE)

Every page must include:

```html
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<title><!-- Unique, descriptive, ≤ 60 chars --></title>
<meta name="description" content="<!-- ≤ 160 chars -->">
<link rel="canonical" href="<!-- absolute URL -->">

<!-- Open Graph -->
<meta property="og:title" content="">
<meta property="og:description" content="">
<meta property="og:image" content="<!-- absolute URL, 1200×630 -->">
<meta property="og:url" content="">
<meta property="og:type" content="website">

<!-- Twitter Card -->
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:title" content="">
<meta name="twitter:description" content="">
<meta name="twitter:image" content="">
```

`robots.txt` and `sitemap.xml` are required in `public/`.

### VIII. Security (NON-NEGOTIABLE)

- **Content Security Policy** is defined in a `<meta http-equiv="Content-Security-Policy">` tag (GitHub Pages does not allow custom response headers by default). CSP must include at minimum: `default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; font-src 'self'; object-src 'none'; base-uri 'self'; form-action 'self'`.
- No inline `<script>` or `style=""` attributes (both violate CSP `'unsafe-inline'`).
- Subresource Integrity (`integrity` + `crossorigin`) is required for any CDN-loaded resource.
- Secrets, API keys, and tokens must **never** appear in committed code or `public/`. Use GitHub Actions secrets for CI-only values.
- Dependency audit (`npm audit`) must pass with zero high/critical findings before any deploy.

### IX. GitHub Pages Deployment (NON-NEGOTIABLE)

Deployment is **CI-only** — no manual `gh-pages` pushes:

```yaml
# .github/workflows/deploy.yml
name: Deploy
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: npm
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck
      - run: npm test
      - run: npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist

  deploy:
    if: github.ref == 'refs/heads/main'
    needs: build
    environment:
      name: github-pages
      url: ${{ steps.deploy.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/deploy-pages@v4
        id: deploy
```

Rules:
- PRs trigger build + lint + test but **do not deploy**.
- Only commits to `main` trigger a deploy.
- The `dist/` folder is **never committed** to the repository.
- A custom domain uses a `CNAME` file in `public/`; HTTPS is enforced via the GitHub Pages settings checkbox.

### X. Code Quality & Tooling (NON-NEGOTIABLE)

All of the following must pass as CI gates; local pre-commit hooks enforce the same:

| Check | Command | Failure blocks |
|---|---|---|
| TypeScript typecheck | `npm run typecheck` | PR merge |
| ESLint | `npm run lint` | PR merge |
| Stylelint | `npm run lint:css` | PR merge |
| Prettier | `npm run format:check` | PR merge |
| Unit tests | `npm test` | PR merge |
| npm audit | `npm audit --audit-level=high` | Deploy |
| Lighthouse CI | `npm run lhci` | Deploy |

`package.json` scripts must include:

```json
{
  "scripts": {
    "dev": "vite",
    "build": "tsc --noEmit && vite build",
    "typecheck": "tsc --noEmit",
    "preview": "vite preview",
    "test": "vitest run",
    "test:watch": "vitest",
    "lint": "eslint src --ext .ts",
    "lint:css": "stylelint \"src/styles/**/*.css\"",
    "format": "prettier --write .",
    "format:check": "prettier --check .",
    "lhci": "lhci autorun"
  }
}
```

### XI. Git Workflow

- `main` is the production branch — it is always deployable.
- Feature branches use the format `feature/<short-slug>` or `fix/<short-slug>`.
- Commits follow [Conventional Commits](https://www.conventionalcommits.org/): `type(scope): description`. Types: `feat`, `fix`, `style`, `refactor`, `test`, `chore`, `docs`.
- Pull Requests require at least one review approval before merge.
- Squash-merge is the default merge strategy to keep `main` history linear.
- **No force-push to `main`.**

### XII. Simplicity & YAGNI

- Write the simplest code that satisfies the requirement. Do not design for hypothetical future requirements.
- Three similar lines of code is preferable to a premature abstraction.
- No half-finished features — every merged commit leaves the site in a fully working, deployable state.
- Comments explain **why**, never what. If what the code does is unclear from reading it, rename identifiers.

## Quality Gates

A pull request may not be merged unless:
1. CI pipeline is green (all checks pass).
2. Lighthouse CI scores meet the defined thresholds.
3. Manual accessibility spot-check passes for any UI change (axe DevTools or similar).
4. At least one peer review approval is recorded.

A deploy to production may not proceed unless:
1. All PR quality gates passed.
2. `npm audit` shows zero high/critical vulnerabilities.

## Governance

This constitution supersedes all other coding standards, preferences, or conventions within this repository. In any conflict, the constitution wins.

Amendments require:
1. A PR with a written rationale for why the current rule fails.
2. Review and approval by the repository owner.
3. All existing code brought into compliance before or alongside the amendment.

No AI agent, tool, or automated process may weaken a security, accessibility, or performance rule without explicit human approval.
